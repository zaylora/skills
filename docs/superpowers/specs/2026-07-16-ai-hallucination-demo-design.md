# “AI 为什么会产生幻觉”口播成片试用设计

## 目标

用 `chengfeng-videocut-skills:口播成片` 完整制作一条约 20 秒的中文竖屏试片，验证以下链路：

1. 从视频与字幕生成 HTML 分镜核对页。
2. 分镜确认后生成可播放的时间线预览。
3. 时间线确认后导出带音频的最终 MP4。
4. 使用 `ffprobe` 和关键帧抽检完成验收。

成功标准：最终视频为 3:4、30fps、有中文口播音频，四段小黑动画与对应口播句同步，画面无溢出、遮挡、浏览器控件或审核界面。

## 假设与边界

- 用户已选择完整成片，并批准主题与四段视觉方向。
- 这是 Skill 试用样例，不追求真人出镜、复杂剪辑或专业配音。
- 使用本机中文 TTS 生成口播音频，再用 FFmpeg 生成 `source_cut.mp4`。
- 实际音频时长是时间线真相源；字幕时间在音频生成后按实际时长校准。
- 按 Skill 门槛执行：分镜核对页和时间线预览仍需分别确认，不能直接跳到导出。

## 口播稿

> AI 有时会一本正经地说错话，这就是我们常说的幻觉。因为它首先是在预测下一个最可能出现的词，不是在数据库里查事实。信息不够时，它也可能把合理的碎片拼成一个看似完整的答案。所以重要结论，要让它给出出处，再自己核对原文。

## 分镜设计

### 1. 流畅不等于正确

- 目标时间：约 0-4 秒。
- 口播：AI 有时会一本正经地说错话，这就是我们常说的幻觉。
- 画面任务：建立问题。
- 画面类型：文字 + 小黑动画。
- 动作：答案卡快速写满；小黑停下并举起问号，强调“流畅不等于正确”。

### 2. 预测下一个词

- 目标时间：约 4-9 秒。
- 口播：因为它首先是在预测下一个最可能出现的词，不是在数据库里查事实。
- 画面任务：解释生成机制。
- 画面类型：HTML 动画。
- 动作：词块沿传送带依次出现；小黑选择高概率词；数据库图标退到背景。

### 3. 用猜测补齐缺口

- 目标时间：约 9-14 秒。
- 口播：信息不够时，它也可能把合理的碎片拼成一个看似完整的答案。
- 画面任务：解释错误如何形成。
- 画面类型：HTML 动画。
- 动作：拼图保留两个空位；小黑放入黄色“猜测块”；完整图形出现但边缘标红。

### 4. 核对来源

- 目标时间：约 14-20 秒。
- 口播：所以重要结论，要让它给出出处，再自己核对原文。
- 画面任务：给出可执行结论。
- 画面类型：文字 + 小黑动画。
- 动作：答案进入来源筛子；无来源内容落下；保留原文卡片并出现绿色核对标记。

## 数据流

1. TTS 生成中文 WAV。
2. FFmpeg 把音频与竖屏基础画面合成为 `source_cut.mp4`。
3. `ffprobe` 获取实际时长，生成与音频匹配的 `subtitles.srt`。
4. 根据字幕语义生成 `storyboard-audit-v1.html`。
5. 分镜确认后创建四个 HTML 动画模块和时间线预览。
6. 时间线确认后创建 `final-player.html`，逐帧截图并用 FFmpeg 合成音频。

## 失败处理

- 本机中文 TTS 不可用：停止制作并报告缺失语音能力，不生成无声成片冒充完成。
- 实际语音时长偏离 20 秒：保留完整口播，以实际时长重新分配字幕和 cue，不机械加速。
- Playwright 或浏览器不可用：分镜阶段可以继续；进入时间线或导出前补齐依赖并重新验证。
- HTML 模块在 1080x1440 逻辑画布中溢出：先修模块固定坐标和 render mode，再进行整片导出。

## 验证

- 素材阶段：`ffprobe` 检查视频时长、1080x1440 基础尺寸、30fps 和音频流。
- 分镜阶段：浏览器打开 HTML，检查四段口播、时间、画面任务、素材来源和镜头动作完整。
- 时间线阶段：使用支持 HTTP Range 的预览服务，检查 seek、播放、音量、倍速和音画同步。
- 导出阶段：检查最终 1620x2160、30fps、音频存在且时长与源视频一致。
- 视觉验收：至少抽取四张对应各模块的关键帧，并与时间线预览同一时刻对照。

## 非目标

- 不实现多种动画风格或配置面板。
- 不加入背景音乐、复杂字幕特效、调色或多轨混音。
- 不修改已安装 Skill 的规则、模板或脚本。
